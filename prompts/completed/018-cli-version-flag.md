<summary>

- Add top-level `--version` / `-V` flag to all three semantic-search entry points (`semantic-search`, `semantic-search-mcp`, `semantic-search-http`)
- Flag prints `<progname> v<__version__>` to stdout and exits 0, BEFORE any subcommand parsing
- `__version__` comes from existing `semantic_search._version.__version__` (hatch-vcs-derived; already importable)
- Closes a convention gap discovered during spec 002-relax-python-floor AC #8 verification (operator had to use `uv tool list` as substitute)
- No new dependencies; pure stdlib + existing version module
- Adds tests covering each entry point's `--version` path

</summary>

<objective>

After this change, running `semantic-search --version`, `semantic-search-mcp --version`, or `semantic-search-http --version` (or the short `-V` form) prints `<binary-name> v<version>` (e.g. `semantic-search v0.15.2`) to stdout and exits with code 0. The flag is recognized BEFORE subcommand dispatch — so `semantic-search --version` does not get rejected as an unknown subcommand. All existing subcommand and flag behavior remains unchanged.

</objective>

<context>

Three entry points are declared in `pyproject.toml:32-35`:

```toml
[project.scripts]
semantic-search-mcp = "semantic_search.__main__:main"
semantic-search-http = "semantic_search.http_server:main"
semantic-search = "semantic_search.__main__:main_cli"
```

Current behavior (all three reject `--version`):
- `semantic-search --version` → "Error: unknown subcommand '--version'" + exit 1 (`main_cli` in `src/semantic_search/__main__.py:91`)
- `semantic-search-mcp --version` → "Error: no subcommand given" or similar (`main` in `src/semantic_search/__main__.py:12`)
- `semantic-search-http --version` → argparse error (uses argparse with no version action; `src/semantic_search/http_server.py:273-281`)

Version source:
- `src/semantic_search/_version.py` is generated by hatch-vcs at build time. The module exposes `__version__: str` and `version: str` (same value).
- Already importable as `from semantic_search._version import __version__` — confirmed by `uv run python -c "from semantic_search._version import __version__; print(__version__)"` → `0.15.3.dev3+gc1a50d7fa`.
- For a release build the version is the clean tag (e.g. `0.15.2`); for dev builds it has a `dev` suffix. Both shapes are valid; print whatever the module reports verbatim.

Why before subcommand parsing:
- `main_cli` currently checks `sys.argv[1]` and treats anything that's not `search` / `duplicates` as an error. A `--version` short-circuit must happen BEFORE that check, otherwise the flag is consumed as an unknown subcommand.
- Same for `main` (mcp): checks `sys.argv[1]` for `serve`.
- For `http_server.main`: uses argparse. The standard pattern is `parser.add_argument("--version", "-V", action="version", version=f"%(prog)s v{__version__}")`. Argparse handles this natively.

Output format:
- Print to **stdout** (so it composes with shell pipelines and `command --version | grep ...`), not stderr.
- Format: `<progname> v<version>` (e.g. `semantic-search v0.15.2`), matching standard conventional output. `<progname>` should be the actual installed binary name, not the Python module — use `Path(sys.argv[0]).name` for the bash-invoked case.
- Exit code 0.

Tests live under `tests/` — see existing patterns in `tests/test_cli.py` (if it exists) or the closest test module to the entry point. Use `pytest` (already in dev deps). For the `--version` path, invoke via `subprocess.run([sys.executable, "-m", "semantic_search", "--version"], capture_output=True, text=True)` or by monkeypatching `sys.argv` and catching `SystemExit`.
