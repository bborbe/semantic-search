**Why three pin sites, not one:**

`pyproject.toml` declares three matched pins for Python 3.14 â€” `[project] requires-python`, `[tool.ruff] target-version`, `[tool.mypy] python_version`. The spec's Non-goal section locks these in lockstep: do not lower the package floor without also moving the linter / typechecker floor (or vice versa). Lowering `requires-python` alone would let 3.13 install but never get linted; lowering the linter alone would lint against 3.13 features on a 3.14-only project (silly). All three move together.

**Why `uv lock` is mandatory after the `pyproject.toml` edit:**

`uv.lock:3` mirrors `requires-python` from `pyproject.toml`. The Acceptance Criteria pin `uv.lock`'s `requires-python = ">=3.13"` and require the lockfile to be regenerated against the new floor (transitive dependency versions may shift if 3.13 cannot consume what 3.14 preferred). Regeneration must happen with `--python 3.13` so the resolver is forced to the lower boundary â€” `uv lock` alone uses the current interpreter (3.14), which may pick transitives that 3.13 cannot resolve.

**Why no source code change:**

The codebase uses only `from typing import Any` + `from __future__ import annotations`. No PEP 695 generics, no `match`/`case`, no 3.14-only stdlib. The spec's Non-goals section explicitly forbids touching `src/` or `tests/`. If `make typecheck` or `make lint` surfaces a new genuine issue at the new floor, the recovery is to either fix the genuine issue OR revert the floor (per the spec's Failure Modes table) â€” not to relax the constraints further.

**Why no scenario change:**

Existing scenarios (`scenarios/001-005`) exercise MCP stdio, REST, CLI, and content-fetch behavior. None of them depend on the Python version floor. **Why no plugin JSON change:**

`.claude-plugin/plugin.json` and `.claude-plugin/marketplace.json` version the plugin surface (slash commands, agents, docs, skills). The Python package floor is not a plugin-surface concern. The spec's Non-goals section explicitly forbids touching these.
